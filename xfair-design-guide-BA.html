💡 The Big Idea

XFAIR serves many customers — each with their own brand

Coca-Cola wants red. A tech company wants blue. A healthcare firm wants green. XFAIR's design system is built so the UI can completely change colors for each customer — without rebuilding anything.

3

✓

One system, many brands

XFAIR looks and feels like each customer's product — not a generic off-the-shelf tool.

⚠ Why this guide exists

Some designs have been made using specific XFAIR colors (our orange, dark navy, and peach) as if they are fixed forever. But those are just XFAIR's own brand colors — the default. When designing, every color choice must be made with the question: "What if this customer's primary color is blue?" The design must still work and look intentional.

💡 Why can't we rebrand success/error colors?

Imagine Coca-Cola's deployment — their primary brand color is red. If we allowed rebranding, their "error" messages would be red AND their buttons would be red. Users wouldn't know what's an error and what's an action. Green = success, amber = warning, and red = error must be universal and unchangeable.

📐 The 4 Rules

Rules every design must follow

Before finalising any screen, make sure all four of these hold true.

1

Primary color = actions only

The primary color should only appear on things the user can click or interact with — buttons, links, active states. It should never be used as a decoration or background on large areas.

Think of it like a traffic light. Orange means "go here, do this." If you paint too many things orange, the user doesn't know what to click.
2

Secondary color = navigation frame only

The dark sidebar/navigation color lives only in the app's chrome — the sidebar, the header strip. It should never bleed into the content area as a text color or button color.

The sidebar is like the walls of a room. Content (headings, body text) always uses dark charcoal, not the wall color.
3

Accent = gentle highlights only

The soft peach/accent color is a whisper, not a shout. It's only used as a very light background tint — for selected rows, hover states, and KPI card backgrounds.

Never use the accent color on text, on buttons, or as a border. It's a background tint, nothing more.
4

Test with a different brand color

Before a design is final, mentally swap the primary color to something very different — like blue or purple. Does the design still look intentional and work well? If yes, it's right.

If anything looks broken or wrong with a different color, it means a hardcoded XFAIR color was used somewhere it shouldn't have been.

✅ Do's & Don'ts

Quick reference

A summary of what to always do and what to avoid when making design decisions.

✓ Always do this
  • Use the primary color for all action buttons (Save, Submit, Create, Publish)
  • Use the accent color for selected rows, hover states, and active nav backgrounds
  • Use fixed charcoal for all headings and body text
  • Use fixed gray for labels, helper text, and placeholders
  • Use success/warning/error colors for status messages and badges only
  • Keep cards, forms, and dialogs on a white background
  • Ask yourself: "Would this still work if the primary color was blue?"
✗ Never do this
  • Never use the primary orange as a background for large areas (sidebar, page, table)
  • Never use the secondary (navy) color for buttons, links, or headings
  • Never use the full primary color as a table row hover/selected background
  • Never use success green or error red for anything other than status
  • Never apply any brand color to borders or dividers — those are always light gray
  • Never use the sidebar color anywhere outside the sidebar/navigation
  • Never design assuming the primary color will always be orange
☑ Design Checklist

Before handing off any screen

Run through these 6 questions on every design. If any answer is "no", revisit the colors on that screen.

  • 1
    Are all action buttons using the primary color?

    Save, Submit, Create, Publish, and any other primary CTA should have the primary color as their background. Secondary buttons (Cancel, Back) should be white with a primary-colored border.

  • 2
    Is the sidebar background the only place the secondary color appears?

    The dark navigation color should only be in the sidebar and top header area. It should not appear on headings, text, or any content inside the main page area.

  • 3
    Are selected rows and hover states using the accent (soft/light) color — not the full primary?

    Table row selection and hover should use the pale accent tint, not the strong primary color. The primary is for buttons and links — not for highlighting rows.

  • 4
    Are page titles and section headings in the fixed dark charcoal?

    Headings should never be the sidebar color or the primary color. They use the fixed dark charcoal (#1F2937) which works regardless of what brand colors a customer has.

  • 5
    Are success/warning/error messages using their fixed colors?

    Green badges always mean success. Amber always means pending/warning. Red always means error. These must never be replaced with a customer's brand color, even if their brand color happens to be one of those.

  • 6
    If I imagine this screen with blue as the primary color, does it still look right?

    This is the ultimate test. Mentally replace all orange with blue. If anything looks strange or out of place, it means a color was hardcoded as "orange" rather than "primary action color." Find it and fix it.
